# 持仓建议与每日持仓 Workbook 设计

> 状态：待用户复核
> 日期：2026-04-23
> 主题：为 `daily_workflow` 增加独立的持仓建议模块，并通过每日自动创建的 Excel workbook 接收人工录入的持仓与原始推荐

## 1. 背景

当前系统已经能完成两类工作：

- `Phase 1 / Phase 2`：生成当日新开仓推荐
- `Phase 3`：盘中监控新品种信号，并在 watchlist 上做轻量重评

但它仍然缺少一条独立的“持仓管理”链路。这会带来两个实际问题：

1. 昨天推荐过的品种，今天如果不再满足“新开仓推荐”门槛，就会直接从推荐结果中消失
2. 对已经持有的仓位，系统没有单独回答“继续持有 / 上移止损 / 减仓 / 平仓”的问题

这会造成用户体验上的错位：

- “今天不推荐新开仓”
- 不等于
- “昨天已经持有的仓位应该离场”

因此，本次设计的目标不是再给现有推荐补几行文案，而是新增一条与“新开仓推荐”并行的“持仓建议”链路。

## 2. 设计目标

本次设计必须解决以下问题：

- 用一份人工可维护的 Excel workbook 记录每日持仓与对应的原始推荐
- 在 `daily_workflow` 中新增一段独立的持仓建议流程
- 对每笔持仓同时展示两套信息：
  - 原始计划
  - 当前盘面重评
- 输出明确的持仓结论：
  - `继续持有`
  - `上移止损`
  - `减仓观察`
  - `平仓`
  - `计划失效待观察`
- 终端只打印真正需要用户即时关注的事件
- 详细比对结果全部写入日志，便于排查“昨天推荐、今天不推荐”的原因

## 3. 非目标

本次明确不做：

- 券商 API 接入或账户权限开通
- 自动下单
- 自动反查历史日志来猜测“最接近的原始推荐”
- 组合级资金管理或仓位优化
- 改写现有 `Phase 1 / Phase 2 / Phase 3` 的核心推荐逻辑

原始推荐由用户手工录入到 workbook 中，不再尝试由系统自动推断。

## 4. 总体方案

推荐方案是新增独立模块 `holdings_advice`，由 `daily_workflow` 在主流程中调用。

模块职责明确分层如下：

- `holdings_advice.loader`
  - 读取当天 workbook
  - 解析 `持仓录入` 与 `原始推荐` 两张表
  - 按 `H001-持仓` / `H001-推荐` 的前缀归并为同一笔交易上下文
  - 对方向字段做标准化

- `holdings_advice.analyzer`
  - 读取一笔持仓、对应原始计划和最新 `daily_df / live_df`
  - 生成两套视角：
    - `原始计划视角`
    - `当前重评视角`
  - 输出持仓结论、建议原因和终端是否需要提醒

- `holdings_advice.presenter`
  - 生成详细日志文本
  - 生成终端精简提醒文本

- `daily_workflow`
  - 保持现有 `Phase 0 / 1 / 2 / 3` 语义不变
  - 追加新的 `Phase 4: 持仓建议`
  - 只负责调度，不承载具体持仓分析细节

## 5. 每日 Workbook 设计

### 5.1 文件路径与命名

每日 workbook 固定创建在：

- [docs/holdings](/Users/yimin.fu/PythonProjects/ctp/docs/holdings)

文件名格式固定为：

- `holdings-YYYY-MM-DD.xlsx`

例如：

- `docs/holdings/holdings-2026-04-23.xlsx`

### 5.2 永久记录 ID

每一笔交易使用一个永久记录 ID，而不是“当天第几行”。

示例：

- `H001-持仓`
- `H001-推荐`

规则如下：

- `H001` 表示同一笔交易的主 ID
- `-持仓` 行记录你的实际持仓
- `-推荐` 行记录你当时实际参考的原始推荐
- 如果第二天对同一品种同方向再次加仓，而且这是基于第二天新的推荐，则视为新交易，使用新的 ID，例如 `H002`
- 已持有仓位在后续每日 workbook 中继续沿用原来的主 ID

### 5.3 Sheet 1：`持仓录入`

字段固定为：

- `记录ID`
- `合约代码`
- `品种名称`
- `方向`
- `手数`
- `开仓均价`
- `开仓日期`
- `备注`

字段说明：

- `方向` 由用户填写 `做多` 或 `做空`
- 系统在读取时兼容 `多/空/long/short` 等别名，但模板说明中统一要求填写 `做多/做空`
- `合约代码 + 方向 + 手数 + 开仓均价` 是最小有效信息

### 5.4 Sheet 2：`原始推荐`

字段固定为：

- `记录ID`
- `推荐日期`
- `推荐时间`
- `合约代码`
- `品种名称`
- `方向`
- `计划类型`
- `原始入场价`
- `原始第一止损位`
- `原始止损位`
- `原始第一止盈位`
- `原始止盈位`
- `原始第一止盈RR`
- `原始准入RR`
- `备注`

字段说明：

- `推荐时间` 保留，因为同一天同一品种可能出现多次盘中重评
- `计划类型` 由用户填写 `趋势` 或 `反转`
- 如果原始推荐只有一档止损，则 `原始第一止损位` 与 `原始止损位` 填相同值
- 一笔实际开出来的仓只保留一条原始推荐，不记录同一天未被采用的其他重评结果

## 6. 每日自动建表

### 6.1 调度时间

自动化任务在每个交易日 `15:00` 运行，负责创建当天 workbook。

### 6.2 创建规则

- 如果当天文件不存在，则创建
- 如果当天文件已经存在，则跳过，不覆盖用户手工内容

### 6.3 续持仓草稿

自动创建时默认启用“续持仓草稿”行为：

- 从前一交易日 workbook 中复制所有非空的 `持仓录入` 行
- 同时复制对应的 `原始推荐` 行
- 保持 `H001` 这类主 ID 不变

这样用户第二天只需要：

- 删除已经平仓的仓位
- 更新仍持仓的手数或均价
- 追加当天新建的 `H00X`

本次实现阶段应更新现有 `每日持仓模板` 自动化，使其输出与本设计一致，而不是继续使用旧字段版本。

## 7. `daily_workflow` 接入方式

### 7.1 流程位置

`daily_workflow` 的执行顺序调整为：

1. `Phase 0`
2. `Phase 1`
3. `Phase 2`
4. `Phase 3`（如果开启盘中监控）
5. `Phase 4: 持仓建议`

说明：

- `Phase 0 / 1 / 2 / 3` 继续只负责“新开仓推荐”和“新品种盘中信号”
- `Phase 4` 负责已持仓仓位的建议
- `--no-monitor` 模式下也应至少执行一轮 `Phase 4`
- 如果开启盘中监控，则 `Phase 4` 在初始分析后执行一次，并在后续每轮监控中同步重评持仓，但终端仍只打印动作变化或关键价位变化

### 7.2 数据来源

`Phase 4` 对每笔持仓读取：

- 当天 workbook 中的 `持仓录入`
- 当天 workbook 中对应的 `原始推荐`
- 最新行情生成的 `daily_df / live_df`

当前重评不重跑 `Phase 1`，只复用已有的技术分析入口，以保持“持仓建议”与“新开仓推荐”边界清晰。

## 8. 持仓分析逻辑

### 8.1 双视角输出

每笔持仓都必须同时输出两套计划：

#### 原始计划视角

- `原始入场价`
- `原始第一止损位`
- `原始止损位`
- `原始第一止盈位`
- `原始止盈位`
- `原始第一止盈RR`
- `原始准入RR`

#### 当前重评视角

- `当前入场价`
- `当前第一止损位`
- `当前止损位`
- `当前第一止盈位`
- `当前止盈位`
- `当前第一止盈RR`
- `当前准入RR`

如果当前重评只产出一档止损，则：

- `当前第一止损位 = 当前止损位`

### 8.2 持仓结论

持仓结论固定为以下五类之一：

- `继续持有`
- `上移止损`
- `减仓观察`
- `平仓`
- `计划失效待观察`

### 8.3 判定规则

#### `继续持有`

触发条件：

- 原始计划和当前重评都未失效
- 当前价未触发原始硬止损或当前重评止损
- 当前方向结构仍然成立

补充说明：

- 即使某品种今天不再属于“新开仓推荐”，也不能仅因此把已持仓结论降级为负面信号

#### `上移止损`

触发条件：

- 当前重评仍支持原持仓方向
- 当前止损位优于原始止损位
- 且变化幅度超过最小提醒阈值

方向规则：

- 做多：`当前止损 > 原始止损`
- 做空：`当前止损 < 原始止损`

#### `减仓观察`

触发条件之一：

- 原始第一止盈位已到达或接近
- 当前盈亏比明显恶化
- 结构仍成立，但信号强度下降

这类结论表示仓位管理需要动作，但不直接等同于平仓。

#### `平仓`

触发条件之一：

- 当前价触发原始硬止损
- 当前重评明确判定结构失效，且方向不再成立

说明：

- “今天没有新推荐”不能作为平仓条件
- `平仓` 必须由价格或结构失效触发

#### `计划失效待观察`

典型场景：

- 原始计划的入场区已经远离
- 原始止盈位已全部达到，旧计划自然完成
- 当前方向仍可能成立，但已经不是原始那套交易计划

该结论用于明确区分：

- 原始计划已完成或失效
- 与
- 当前方向反了

二者不是一回事。

### 8.4 关键价位变化阈值

为了避免终端刷屏，只有在关键价位变化明显时才提醒。

默认提醒阈值为：

- `max(最小跳动 * 3, 0.3%)`

适用字段：

- 止损位
- 第一止盈位
- 止盈位

## 9. 日志与终端输出

### 9.1 日志详细版

日志新增固定段落：

- `=== Phase 4: 持仓建议 ===`

每笔持仓固定输出：

1. 持仓基本信息
2. 原始计划
3. 当前重评
4. 持仓结论
5. 建议原因

示例格式：

```text
H001 生猪 做多 2手 开仓均价11480 当前价11540
原始计划: 入场11490 第一止损11160 止损11142 第一止盈12265 止盈12445 第一止盈RR 2.22 准入RR 2.74
当前重评: 入场11545 第一止损11165 止损11151 第一止盈12265 止盈12445 第一止盈RR 1.83 准入RR 2.28
持仓建议: 继续持有
建议原因: 原始趋势未破坏，但当前准入RR下降，暂不建议加仓
```

日志末尾增加摘要：

- `持仓摘要: 继续持有2 上移止损1 减仓观察0 平仓1 计划失效1`

### 9.2 终端精简版

终端只打印以下两类事件：

- 新品种首次出现可操作信号
- 已持仓的建议发生动作变化或关键价位显著变化

具体包括：

- 第一次出现 `平仓`
- 第一次出现 `上移止损`
- 第一次出现 `减仓观察`
- 从 `继续持有` 变成 `计划失效待观察`
- 止损或止盈位变化超过阈值

终端提醒必须同时带出关键价格，不能只输出一个抽象结论。

其余持仓信息只写入日志，不进入终端。

## 10. 异常处理

### 10.1 当天 workbook 不存在

行为：

- 不影响 `Phase 0 / 1 / 2 / 3`
- `Phase 4` 跳过
- 日志记录：`未找到今日持仓文件，跳过 Phase 4`

### 10.2 Excel 字段缺失或格式错误

行为：

- 只跳过出错的那一条 `Hxxx`
- 日志明确指出：
  - 哪个 `记录ID`
  - 哪一列有问题
  - 期望格式是什么

原则：

- 单行脏数据不能让整个 workflow 失败

## 11. 测试要求

实现阶段至少覆盖以下场景：

- 当天 workbook 自动创建成功
- 当天 workbook 已存在时不会覆盖
- 前一日持仓能正确续写到新 workbook
- `H001-持仓` 与 `H001-推荐` 能正确归并
- 方向字段标准化正确
- workbook 缺列或单行脏数据时仅局部报错
- `Phase 4` 能在 `--no-monitor` 模式下运行
- `继续持有 / 上移止损 / 减仓观察 / 平仓 / 计划失效待观察` 五类结论均有测试
- 终端只在动作变化或关键价位变化时打印
- 详细结果全部进入日志

## 12. 实施约束

本次实现应保持以下边界：

- 不把持仓逻辑继续堆进 `daily_workflow.py`
- 优先复用现有技术分析和盘中数据获取能力
- 原始推荐只信 workbook 人工录入，不做自动猜测
- 现有“每日持仓模板”自动化需要更新为本设计，而不是并行创建第二套模板

## 13. 结论

本设计将“新开仓推荐”和“已持仓建议”明确拆成两条平行链路：

- 新开仓推荐继续回答“今天能不能新开”
- 持仓建议单独回答“昨天已经开的仓今天怎么办”

用户每天只需维护一份 workbook，就能在同一轮 `daily_workflow` 中同时得到：

- 新品种推荐
- 已持仓建议
- 原始计划与当前重评的并列对比

这既能降低“推荐消失引发的恐慌”，也能为后续排查“为什么前后不一致”提供完整证据链。
