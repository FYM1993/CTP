> 状态：待用户复核

> 主题：将 Phase 2 从“反转信号门控”改为“反转入场 + 趋势延续入场”双路径并行，并同步重写 README 中对 Phase 1/2/3 与回测的策略说明

## 1. 背景

当前项目在职责划分上已经基本清晰：

- `Phase 1`：发现机会，输出关注优先级和机会标签
- `Phase 2`：定方向、生成交易计划
- `Phase 3`：盘中监控、触发入场提示

但当前生产级 `Phase 2` 仍有一个核心缺口：

- 技术分数可以决定 `long / short / watch`
- 真正生成交易计划时，却仍然依赖 `assess_reversal_status(...).has_signal`
- 这意味着当前系统本质上仍是“反转信号门控”
- 还没有一条独立的“趋势延续入场”生产路径

这会带来两个直接问题：

1. 像 `LH0` 这种长期下跌品种，系统可以合理地不去抄底做多，但也缺少顺势做空的独立入口
2. README 当前对“顺势入场”的描述早于真实实现，用户从文档中看到的策略和生产代码不完全一致

因此本次改造的目标不是再加一层文案，而是把 `Phase 2` 的双路径逻辑真正落地，并把 README 改成“当前真实策略说明”。

## 2. 设计目标

本次设计解决两件事：

- 把 `Phase 2` 改成真正的双路径入场引擎
- 把 README 改成一份完整、准确、面向使用者的策略说明

本次必须达到：

- 保留现有反转入场链路
- 新增一条独立的双边趋势延续入场链路
- `long / short` 双边都支持
- 回测和诊断能够区分“反转单”和“趋势单”
- README 能完整说明 `Phase 1 / Phase 2 / Phase 3 / 回测`

本次明确不做：

- `Phase 1` 评分框架重构
- `Phase 3` 大改
- 参数优化
- 组合层资金管理
- 实盘下单闭环

## 4. 总体设计

### 4.1 `Phase 2` 改成双路径并行

`Phase 2` 从本次开始明确拆成两条独立入场路径：

- `reversal_entry`
- `trend_continuation_entry`

两条路径都基于：

- 同一份日线数据
- 同一个方向 `direction`
- 同一组技术评分与支撑阻力计算

但它们各自独立判断“是否值得生成交易计划”。

### 4.2 统一交易计划结构

无论计划来自反转路径还是趋势路径，都统一输出同一种 `trade plan` 结构。

新增字段：

- `entry_family`
  - `reversal`
  - `trend`
- `entry_signal_type`
  - 例如 `Spring`、`SOS`、`Pullback`、`TrendBreak`
- `entry_signal_detail`
  - 供报告、日志、回测和诊断解释

保留字段：

- `direction`
- `entry`
- `stop`
- `tp1`
- `tp2`
- `rr`
- `score`
- `actionable`

### 4.3 双路径择优

最终 `Phase 2` 产出 plan 的规则如下：

1. 先独立评估反转路径
2. 再独立评估趋势延续路径
3. 将所有满足 `score gate + rr gate` 的候选计划收集起来
4. 如果没有候选计划，则 `actionable = False`
5. 如果有一个候选计划，则直接采用
6. 如果两条路径都满足，则优先采用 **盈亏比更高** 的计划

诊断层还要保留未被采用的另一条路径，便于回测统计：

- 为什么被落选
- 两条路径哪一条更常胜出
- 哪类计划最终更有效

## 5. 反转入场路径

### 5.1 保留当前信号体系

反转路径继续沿用当前 Wyckoff/VSA 反转语义，不在本次改造中推翻。

做多目标信号：

- `SC`
- `StopVol_Bull`
- `Spring`
- `SOS`

做空目标信号：

- `BC`
- `StopVol_Bear`
- `UT`
- `SOW`

### 5.2 反转计划的生成条件

反转路径继续使用当前核心原则：

- 有新鲜入场信号
- 方向分数达标
- 盈亏比达标

计划参数保持当前思路：

- `entry = 当前价`
- `stop = 信号K线极值 ± 0.5 ATR`
- `tp1 / tp2` 从支撑阻力与 Fib 目标推导

### 5.3 改造边界

本次不重写 `assess_reversal_status()` 的总体语义，只做必要的字段对齐：

- 返回结果能被统一包装成 `trade plan`
- 报告与回测能正确读取 `entry_family = reversal`

## 6. 趋势延续入场路径

### 6.1 设计原则

趋势延续路径必须独立于反转路径，不再依赖 `reversal["has_signal"]`。

它解决的是：

- 趋势已经确立
- 基本面或技术面支持方向继续
- 希望在回撤或突破中顺势参与

它不解决：

- 顶底反转确认
- 抄底摸顶

### 6.2 双边趋势定义

第一版趋势定义使用克制、可解释的结构，不追求复杂：

做多趋势：

- `MA20 > MA60`
- 近段高点和低点抬升

做空趋势：

- `MA20 < MA60`
- 近段高点和低点下移

### 6.3 双边入场模式

趋势延续路径包含两个子模式：

- `Pullback`
- `TrendBreak`

#### 做多

`Pullback`

- 上涨趋势中
- 价格回踩 `MA20` 附近
- 回踩时量能收缩
- 不跌破最近摆动低点
- 并出现重新转强迹象

`TrendBreak`

- 上涨趋势中
- 放量突破近 `20` 日关键高点

#### 做空

`Pullback`

- 下跌趋势中
- 价格反弹到 `MA20` 附近
- 反弹时量能收缩
- 不突破最近摆动高点
- 并出现重新转弱迹象

`TrendBreak`

- 下跌趋势中
- 放量跌破近 `20` 日关键低点

### 6.4 趋势计划参数

趋势路径的计划参数与反转路径分开定义。

`Pullback`

- 止损放在最近摆动点 `± 0.5 ATR`
- 优先级高于 `TrendBreak`
- 因为通常盈亏比更好

`TrendBreak`

- 止损放在突破前最近反向摆动点 `± 0.5 ATR`

`tp1 / tp2`

- 继续使用现有支撑阻力和 Fib 逻辑
- 不为趋势路径额外引入新的目标位算法

### 6.5 趋势路径门槛

趋势路径仍需通过：

- 方向明确
- `Phase 2` 评分门槛
- 盈亏比门槛

门槛原则：

- 做多：`score > +20`
- 做空：`score < -20`
- `rr >= 1.0`

如果后续回测表明趋势单更适合更高门槛，再在后续版本调整；本次先与当前门槛对齐。

## 7. 回测与诊断改造

### 7.1 回测必须识别两类计划

最小回测框架要能区分：

- 反转单
- 趋势单

因此每笔交易记录新增：

- `entry_family`
- `entry_signal_type`

### 7.2 诊断要能回答的问题

回测输出必须能回答：

- `Phase 2` 当前到底产出了多少反转计划
- `Phase 2` 当前到底产出了多少趋势计划
- 两类计划分别通过了多少次 `score gate`
- 两类计划分别通过了多少次 `rr gate`
- 哪类计划最后真的触发了交易

### 7.3 第一轮验证重点

第一轮验证不要求立刻覆盖全部品种，只要求验证路径成立。

优先验证：

- 找 `LH0` 一段明显反弹区间，验证当前反转做多剧本
- 用明确下跌区间，验证新增趋势延续做空路径

也就是说：

- 策略实现先做双边
- 回测样例第一轮可先测单边 case

## 11. 风险与取舍

### 11.1 为什么不把趋势逻辑继续塞进 `assess_reversal_status()`

因为那会继续扩大语义混乱：

- 反转状态函数不该再承载趋势延续策略
- 回测也很难分清到底是反转单还是趋势单

### 11.2 为什么这次先不改 `Phase 3`

因为当前主要缺口在 `Phase 2` 是否能产出计划。

如果 `Phase 2` 仍不产出趋势单：

- 再怎么调 `Phase 3`
- 也不会解决“顺势做空根本进不了计划层”的问题

### 11.3 为什么 README 要和代码同轮改

因为这次改动直接影响用户对策略的理解。

如果代码先改、README 不同步：

- 会继续出现“代码和文档不一致”
- 回测结果也会变得难解释

## 12. 成功标准

本次改造完成后，应满足：

- `Phase 2` 不再只有反转一条可出计划路径
- 双边趋势延续路径已在生产代码中生效
- 回测和诊断能区分反转单与趋势单
- README 能准确解释当前真实策略与回测方式
- 用户再看 `LH0` 这类案例时，能明确区分：
  - 为什么反转做多没出计划
  - 什么时候趋势做空会出计划
