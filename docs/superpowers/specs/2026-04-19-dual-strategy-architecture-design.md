# 双策略架构拆分设计

日期: 2026-04-19

## 背景

当前系统已经演进出两种不同的交易假设：

1. 基本面反转
   - 典型场景: 生猪、多晶硅这类处于长期低位、接近成本线、存在出清或再平衡预期的品种
   - 核心问题: 是否处于长期失衡区，是否出现反转确认

2. 趋势跟随
   - 典型场景: 明确上涨或下跌趋势中的顺势参与
   - 核心问题: 趋势是否成立，是否出现趋势延续入场

之前的实现把这两种假设混在同一条 `Phase 2` 链路里，只在入场层做 `reversal` / `trend` 区分。这会带来三个问题：

- `Phase 1` 的基本面语义和 `Phase 2` 的趋势延续语义互相污染
- 回测归因变得混乱，无法清楚回答“是反转逻辑有效，还是趋势逻辑有效”
- 继续在同一条 `Phase 2` 上加规则，只会让边界越来越不清晰

因此，系统需要在 `Phase 0` 之后就拆成两套独立策略线，共享基础设施，但不共享策略假设。

## 目标

- 在 `Phase 0` 之后，将系统拆成两条独立策略线：
  - 策略 A: 基本面反转
  - 策略 B: 趋势跟随
- 两条策略线共享行情、日志、报告、回测基础设施
- 两条策略线拥有各自独立的：
  - 选品逻辑
  - 入场逻辑
  - 退出逻辑
  - 回测口径
  - 统计指标
- 最终可以在统一的上层编排中合并输出，但不再在底层共享同一个 `Phase 2` 决策过程

## 非目标

- 本轮不引入自动下单
- 本轮不实现组合层仓位管理
- 本轮不直接删除现有全部单线实现
- 本轮不要求一次性完成两套策略的全部优化

## 核心设计

### 总体分层

新架构分为四层：

1. 共享基础设施层
   - 数据加载
   - 缓存
   - 日志
   - 报告落盘
   - 回测驱动

2. 策略 A: 基本面反转
   - 发现长期失衡机会
   - 等待反转确认

3. 策略 B: 趋势跟随
   - 发现趋势型交易机会
   - 等待趋势延续确认

4. 上层编排层
   - 调用两条策略线
   - 聚合结果
   - 展示和排序

### 策略 A: 基本面反转

策略 A 保留当前系统最核心的原始目标。

#### 选品逻辑

策略 A 的候选池主要由 `Phase 1` 的基本面机会发现给出，重点关注：

- 价格偏离长期均衡
- 接近成本线或显著低于成本线
- 行业深亏或暴利
- 出清压力 / 扩产压力
- 库存、仓单、利润等基本面异常

其结果不是“趋势强弱”，而是“是否值得等待反转”。

#### 入场逻辑

策略 A 的 `Phase 2A` 只做反转确认，不做趋势延续。

多头侧:
- `SC`
- `StopVol_Bull`
- `Spring`
- `SOS`

空头侧:
- `BC`
- `StopVol_Bear`
- `UT`
- `SOW`

方向来自基本面假设和反转确认，不再夹带趋势延续逻辑。

#### 退出逻辑

策略 A 的退出依据应围绕“反转失败”而不是“趋势抖动”：

- 初始止损
- `tp1 / tp2`
- 反转结构失效
- 关键日线反向信号

### 策略 B: 趋势跟随

策略 B 是独立策略，不从基本面反转候选池衍生。

#### 选品逻辑

策略 B 的候选池应由趋势 universe 单独产生，优先考虑：

- 趋势方向明确
- 均线和动量结构清晰
- 波动和流动性足够
- 不处于明显震荡状态

这一步不要求基本面低位或高位，也不以成本线为主锚。

#### 入场逻辑

策略 B 的 `Phase 2B` 只做趋势延续确认：

多头:
- 上涨趋势中的回踩企稳
- 放量突破

空头:
- 下跌趋势中的反弹受阻
- 放量跌破

信号标签可以继续保留 `Pullback` / `TrendBreak`，但它们只属于策略 B。

#### 退出逻辑

策略 B 的退出依据应围绕“趋势失效”：

- 初始止损
- `tp1 / tp2`
- 趋势结构破坏
- 明确反向突破

### 上层编排

`daily_workflow` 不再理解一条“混合 `Phase 2`”。

新的上层流程应是：

1. `Phase 0`
   - 准备共享数据

2. 策略 A
   - `P1A` 基本面反转候选
   - `P2A` 反转入场确认

3. 策略 B
   - `P1B` 趋势候选
   - `P2B` 趋势延续确认

4. 聚合
   - 合并两条策略线的 actionable / watchlist
   - 明确标记 `strategy_family = reversal_fundamental / trend_following`

### 回测拆分

回测必须同步拆分，否则后续归因仍然混乱。

#### 回测 A

只验证“基本面反转策略”的技术执行层：

- 输入品种来自策略 A universe
- 只跑反转入场
- 不跑趋势延续

#### 回测 B

只验证“趋势跟随策略”的技术执行层：

- 输入品种来自策略 B universe
- 只跑趋势延续
- 不跑反转入场

#### 结果要求

回测输出必须能明确区分：

- 这是哪一套策略
- 候选来自哪条策略线
- 入场属于哪套逻辑
- 退出属于哪套逻辑

这样后面才能回答：

- 基本面反转是否有效
- 趋势跟随是否有效
- 哪一套更适合某个品种

## 模块边界

建议按功能拆成如下模块：

- `scripts/strategy_reversal/`
  - `screen.py`
  - `pre_market.py`
  - `intraday.py`
  - `backtest.py`

- `scripts/strategy_trend/`
  - `screen.py`
  - `pre_market.py`
  - `intraday.py`
  - `backtest.py`

- `scripts/shared/`
  - 行情
  - 日志
  - 公共模型
  - 报告工具

- `scripts/cli/daily_workflow.py`
  - 只负责编排，不再承载混合策略判断

## 迁移策略

不直接一刀切重写，按三步迁移。

### 第一步

保留当前基础设施和回测器，只把策略边界拆开：

- 明确策略 A / B 的输入输出
- 在代码里先把“反转链路”和“趋势链路”从单一 `Phase 2` 中剥离

### 第二步

让 `daily_workflow` 同时编排两条策略线：

- A 先跑
- B 再跑
- 最后统一输出

### 第三步

各自独立优化：

- 策略 A 继续围绕反转信号和基本面低位修正
- 策略 B 独立优化趋势选品、趋势入场、趋势退出

## README 要求

README 后续必须按这个新架构重写：

- 策略 A 是什么
- 策略 B 是什么
- 它们分别看哪些指标
- 它们分别怎么回测
- 不再写成“一条策略里既反转又趋势延续”

## 决策

从现在开始，系统架构以“双策略线”为准：

- 基本面反转
- 趋势跟随

后续的指标调优和回测分析，都必须先落在其中一条策略线上，再做局部优化。不得再把两套假设继续混在同一条 `Phase 2` 中演进。
