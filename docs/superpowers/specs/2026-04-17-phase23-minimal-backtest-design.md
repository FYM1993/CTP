# Phase 2/3 最小回测框架设计

> 状态：待用户复核
> 日期：2026-04-17
> 主题：为当前 CTA 工作流引入基于 TqBacktest 的最小回测框架，先验证 Phase 2/3 的执行层有效性

## 1. 背景

当前项目已经具备较完整的研究与监控流程：

- `Phase 1`：基本面机会发现
- `Phase 2`：日线级技术分析、方向选择、止损止盈计划
- `Phase 3`：盘中实时监控与信号提示

但系统仍然缺少一个关键能力：**历史可验证性**。

目前的主要问题不是“能不能发现机会”，而是：

1. `Phase 2/3` 的开仓、止损、止盈规则是否真有统计优势
2. 做多和做空两条链路是否都能在历史上成立
3. 当前系统的 `entry / stop / tp1 / tp2` 在历史上是否合理
4. 后续要不要接真实账户权限，缺乏可量化依据

因此第一阶段不做全流程历史回测，而是先做一个**最小回测框架**，只回测 `Phase 2/3`。

## 2. 设计目标

本次设计只解决一件事：

- 给定一个或多个手工 case，验证 `Phase 2/3` 的执行层在历史上的表现

第一版必须满足：

- 同时覆盖 `long` 和 `short`
- 不依赖真实账户或下单权限
- 尽量复用现有 `Phase 2/3` 逻辑，不重写一套策略
- 能输出逐笔交易记录和基础绩效指标
- 能作为后续系统完善的底座

第一版明确不做：

- `Phase 1` 历史选品回测
- 参数优化
- 组合层资金管理
- 实盘下单联动
- 多账户、多策略组合回测

## 3. 范围边界

### 3.1 回测对象

第一版只回测：

- `Phase 2` 生成的交易计划
- `Phase 3` 触发的入场与退出

也就是说，`Phase 1` 在第一版里被视为外部输入。

### 3.2 输入方式

第一版通过显式 case 输入，不通过全市场扫描：

- `symbol`
- `name`
- `direction`
- `start_dt`
- `end_dt`
- 可选备注

首批基准 case：

- `LH0 long`
- `PS0 short`

### 3.3 输出方式

第一版输出：

- 单个 case 的逐笔交易明细
- 单个 case 的绩效汇总
- 批量 case 的汇总表

## 4. 核心设计原则

### 4.1 忠于当前系统，而不是先发明新策略

第一版回测必须尽量忠于当前系统的实际语义：

- `Phase 2` 用日线判断是否值得做
- `Phase 3` 用 `5m` 结构识别信号
- 盘中并不是等完整 `5m` bar 收盘才判断，而是高频检查正在形成中的 `5m` bar

因此回测不能偷换成“只在完整 `5m` 收盘时做判断”的粗粒度方案。

### 4.2 信号周期与执行时钟分离

第一版采用：

- 信号周期：`5m`
- 执行时钟：`1m`

这意味着：

- 用 `1m` 行情推进时间
- 每分钟重建当前正在形成的 `5m` bar
- 每分钟在这个“进行中的 `5m` bar”上运行 `Phase 3` 信号逻辑

这样既忠于当前系统，又避免把 `Phase 3` 粗暴改造成纯 `1m` 策略。

### 4.3 开仓时固定交易计划

`Phase 2` 的交易计划在第一版中视为开仓时固定：

- `entry_ref`
- `stop`
- `tp1`
- `tp2`

一旦开仓：

- 不再每天动态更新 `stop/tp1/tp2`
- 不在第一版做移动止损
- 不做部分止盈后的重新规划

这样结果更容易解释，也更适合做基准验证。

## 5. 回测语义定义

### 5.1 `entry` 的真实语义

当前系统里的 `entry` 不是“挂单触发价”，而是：

- `Phase 2` 生成计划时的参考入场价
- 主要用于展示、记录、计算盈亏比

它**不是**“后续价格碰到它就自动成交”。

真正触发开仓的是：

- `Phase 3` 的信号

因此第一版回测里：

- `entry_ref` 仅用于记录计划
- 实际成交价由 `Phase 3` 信号触发时的当下市场价格决定

### 5.2 日线计划层

每个交易日开盘前：

- 使用“截至前一交易日收盘可见”的日线数据
- 运行 `Phase 2`
- 如果 `Phase 2` 未给出可操作机会，则当日不进入盘中执行
- 如果 `Phase 2` 给出可操作机会，则生成一份交易计划

交易计划包含：

- `direction`
- `entry_ref`
- `stop`
- `tp1`
- `tp2`
- `phase2_score`
- `reversal_status`

### 5.3 盘中执行层

在存在交易计划且尚未开仓的前提下：

- 用 `1m` 行情逐分钟推进
- 每分钟将当日 `1m` 数据聚合成“当前进行中的 `5m` bar`
- 每分钟运行一次 `Phase 3` 信号识别

一旦出现满足方向的入场信号：

- 在当下 `1m` 价格附近成交
- 建立持仓

### 5.4 持仓管理

开仓后进入持仓状态。

第一版持仓规则：

- 同一 case 同时只允许一笔主仓
- 未平仓前，不重复开同方向新仓
- 不做加仓
- `tp1` 先作为观察指标记录命中，不在第一版触发部分止盈
- 真正平仓条件为：
  - 触发 `stop`
  - 触发 `tp2`
  - 日线逻辑失效

### 5.5 平仓触发

第一版定义三类平仓原因：

1. `stop`
   - 价格触及固定止损位
2. `tp2`
   - 价格触及第二目标位
3. `phase2_invalidated`
   - 次日开盘前重新跑 `Phase 2` 后，方向不再成立

`tp1` 仅记录是否命中，用于分析：

- 计划目标是否常被提前兑现
- 后续是否值得引入“到 `tp1` 减仓/提保本”机制

## 6. Long/Short 链路覆盖

第一版必须同时覆盖多空两条链路。

### 6.1 `Phase 2` 多空差异

当前系统中，`long/short` 会影响：

- 方向评分解释
- 反转信号类型
- 止损位置
- 止盈目标的搜索方向
- 盈亏比计算

当前多空共用的一点是：

- 参考入场价都取当前价

### 6.2 `Phase 3` 多空差异

当前系统中，`Phase 3` 的盘中信号对多空有镜像分支：

- `long`
  - 开多 / 加多 / 平多
- `short`
  - 开空 / 加空 / 平空

第一版回测必须保留这种方向差异，不做统一抽象后丢失语义。

## 7. 模块设计

第一版建议拆成以下模块。

### 7.1 `scripts/backtest_models.py`

职责：

- 定义回测数据结构

建议包含：

- `BacktestCase`
- `TradePlan`
- `TradeRecord`
- `BacktestResult`

### 7.2 `scripts/backtest_cases.py`

职责：

- 管理回测 case

第一版支持：

- 内置 case 注册表
- 通过 `case_id` 选择案例

### 7.3 `scripts/backtest_phase23.py`

职责：

- 实现 `Phase 2/3` 最小回测引擎

核心能力：

- 日线计划生成
- `1m` 推进
- 当前 `5m` bar 聚合
- 信号触发
- 持仓推进
- 平仓记录

### 7.4 `scripts/backtest_metrics.py`

职责：

- 对交易记录做统计

输出：

- 总交易数
- 胜率
- 平均收益
- 总收益
- 最大回撤
- `tp1/tp2/stop` 命中率

### 7.5 `scripts/run_backtest.py`

职责：

- 命令行入口

第一版示例：

- `python scripts/run_backtest.py --case lh0_long`
- `python scripts/run_backtest.py --case ps0_short`

## 8. 数据流设计

每个 case 的回测流程如下：

1. 读取 case 配置
2. 用 `TqBacktest` 加载回测区间数据
3. 每个交易日前，基于历史日线运行 `Phase 2`
4. 若生成交易计划，则进入当日盘中扫描
5. 用 `1m` 行情重建当前 `5m` bar
6. 在每分钟时点运行 `Phase 3` 信号
7. 触发开仓后记录真实成交价
8. 后续按固定 `stop/tp2` 与日线失效规则管理退出
9. 生成交易记录与统计结果

## 9. 成交模型

第一版成交模型保持保守但尽量贴近当前系统：

- 入场成交价：信号触发当下的 `1m` 价格
- 止损/止盈触发价：触及阈值的当下 `1m` 价格

第一版不引入：

- 滑点模型
- 手续费模型
- 盘口撮合模型
- 挂单排队模型

这些都留到后续版本。

## 10. 指标输出

### 10.1 逐笔交易明细

至少包含：

- case id
- 品种
- 方向
- 开仓时间
- 开仓价
- 平仓时间
- 平仓价
- 退出原因
- 持仓分钟数
- 持仓天数
- 计划止损
- 计划 `tp1`
- 计划 `tp2`
- 是否命中 `tp1`
- 单笔收益率

### 10.2 case 汇总

至少包含：

- 总交易数
- 胜率
- 平均单笔收益
- 总收益率
- 最大回撤
- 平均持仓时长
- `tp1` 命中率
- `tp2` 命中率
- `stop` 触发率

### 10.3 信号归因

第一版建议同步记录：

- 开仓来源于哪类 `Phase 3` 信号
- 当时的 `Phase 2` 总分
- `reversal_status.signal_type`

这样后续可以直接做消融分析。

## 11. 测试策略

第一版必须覆盖：

1. case 解析与加载
2. `1m -> 进行中 5m bar` 聚合正确性
3. `long` 与 `short` 链路都能建立交易计划
4. 开仓后 `stop / tp2 / 日线失效` 三类退出都能触发
5. `tp1` 命中记录正确
6. 汇总统计结果正确

测试层次：

- 单元测试：数据结构、聚合、指标统计
- 集成测试：单 case 回测闭环

## 12. 风险与限制

第一版已知限制：

- 不包含 `Phase 1` 历史选品
- 不包含真实成交滑点和手续费
- 不包含部分止盈与移动止损
- 不包含多仓位、多品种组合风控

因此第一版回测结果的用途是：

- 判断策略是否值得继续研发
- 找出多空两条链路中最明显的问题

而不是：

- 直接作为真实资金收益预估

## 13. 后续系统完善路线图

最小回测框架落地后，系统按以下顺序增强。

### 阶段 2：退出逻辑闭环

- `tp1` 部分止盈
- 止损提保本
- 动态移动止损
- 反向强信号退出

### 阶段 3：归因与消融

- `Phase 2` 单独回测
- `Phase 2 + Phase 3` 对比
- Wyckoff / VSA / OI / 经典指标分层归因

### 阶段 4：纳入 Phase 1

- 引入历史 `Phase 1` 输出
- 回测“选品 + 执行”的联动效果

### 阶段 5：组合与风控

- 同产业链去重
- 单笔风险统一
- 总风险预算
- 相关性约束

## 14. 推荐结论

第一版最小回测框架采用：

- `TqBacktest`
- case 驱动
- `Phase 2` 日线计划
- `Phase 3` `5m` 信号
- `1m` 推进与成交
- 开仓时固定 `stop/tp1/tp2`

这是当前项目最小、最稳、最能尽快产生有效验证结果的方案。
